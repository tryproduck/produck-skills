## 1. Problem Statement

Enter Pro is a powerful AI coding agent that accelerates development velocity. However, it operates as a **black box** — making changes across files with no safety layer, no rollback mechanism, no staged review, and no persistent memory. As developer trust in the agent increases, so does the blast radius when it gets something wrong.

**The core problem:**
> Developers using Enter Pro have no reliable way to *audit*, *control*, *recover from*, or *persist knowledge across* agent-driven coding sessions.

**Affected user:** Developers using Enter Pro for real project work in multi-file environments.

**Cost of the status quo:** Each failed or destructive agent session costs recovery time via additional tokens. Context loss between sessions costs 5–10 minutes per session re-explaining conventions. Combined, this erodes the 10x productivity promise.

---

## 2. Goals

| Goal | Metric |
|------|--------|
| Reduce recovery time from bad agent sessions | < 30 seconds via one-click rollback |
| Eliminate context drift between sessions | 100% of manual file edits surfaced before next agent prompt |
| Enable selective acceptance of agent changes | Per-file accept/reject before any commit |
| Prevent agent work on broken environments | Health score computed before every session |
| Eliminate per-session re-teaching | Zero re-teaching for stored project rules |

---

## 3. Scope

### In Scope
- Atomic Git-backed checkpoint system (save state before agent runs)
- Git diff-based memory sync detector (surface manual edits to the agent)
- Per-file accept/reject diff review panel
- Workspace health audit (git, node_modules, npm audit, .env)
- Persistent project rules memory (cross-session agent directives via TrueMemory SQLite)

### Out of Scope
- Changes to Enter Pro's core language model or prompt engine
- Cloud sync of checkpoints across machines
- Real-time collaboration features

## 4. User Stories

### FP1 — Checkpoint & Rollback
> *Produck Feedback ID: `8c2df628-1c60-460f-837e-8e7eb6b5a79d` — Captured 2026-06-22T00:15:15Z*

> *As a developer, I want to save a named snapshot of my workspace before every agent session, so that if the agent makes destructive changes I can restore everything in one click.*

**Acceptance Criteria:**
- User can save a labeled checkpoint from inside the IDE
- Checkpoint is backed by a real git commit with the workspace hash stored
- User can restore any checkpoint with a single button click
- Restore hard-resets the git working tree to the checkpoint state
- Operation completes in under 5 seconds

---

### FP2 — Memory Sync
> *Produck Feedback ID: `b7a6931b-c1eb-4892-9189-4c759ea87348` — Captured 2026-06-22T00:33:38Z*

> *As a developer, I want the agent to be aware of any manual file edits I made between sessions, so it doesn't overwrite my changes with stale assumptions.*

**Acceptance Criteria:**
- System detects all files changed since the last agent interaction (via `git diff --numstat HEAD`)
- Drifted files are listed in a clear panel showing additions and deletions
- A "Copy Sync Prompt" button generates a structured context update for the agent
- Prompt explicitly instructs the agent to discard cached versions and use the provided content

---

### FP3 — Diff Review (Staged Accept/Reject)
> *Produck Feedback ID: `adc70843-f2fc-4207-91e2-f6102b62b9d0` — Captured 2026-06-22T00:38:31Z*

> *As a developer, I want to review every file the agent changed before it gets committed, and accept or reject changes on a per-file basis.*

**Acceptance Criteria:**
- After an agent session, all modified files are listed with `+additions / -deletions` counts
- User can open a diff view for each file
- User can "Accept" a file (keep changes) or "Reject" a file (restore from last checkpoint)
- "Accept All" and "Reject All" bulk operations are available
- Status of each file (accepted/rejected/pending) is tracked visually

---

### FP4 — Workspace Health Check
> *Produck Feedback ID: `4ac21a1b-762e-4ef0-a170-046461197f1b` — Captured 2026-06-22T00:43:08Z*

> *As a developer, I want a pre-flight workspace audit before starting an agent session, so the agent doesn't write code on a broken foundation.*

**Acceptance Criteria:**
- Health check runs `git`, `npm ls`, `npm audit`, and checks for `.env` in parallel (completes in < 6 seconds)
- Results shown as a scored gauge (0–100%) with per-item pass/fail/warn breakdown
- Score thresholds: ≥85% = Healthy, 60–84% = Degraded, <60% = At Risk
- Actionable recommendations listed for each failing check

---

### FP5 — Persistent Memory (Long-Term Rules)
> *Produck Feedback ID: `e6489944-0e48-446d-af7c-6070c534c6b1` — Captured 2026-06-22T00:56:32Z*

> *As a developer, I want to store project-specific rules and coding conventions that the agent automatically receives at the start of every session, so I never have to re-teach it.*

**Acceptance Criteria:**
- Developer can add/remove named project directives (e.g. "Always use arrow functions")
- Rules are persisted in a local SQLite database via TrueMemory
- "Copy Recall Prompt" generates a single structured prompt with all rules to inject into the agent
- Rules persist across IDE restarts and agent sessions

## 5. Implementation Phases

### Phase 1 — Foundation (Checkpoint + Health Check)
**Dependency:** Git initialized in workspace
**Scope:** `checkpoint.ts`, `healthCheck.ts`, dashboard Checkpoints + Health tabs
**Verification:** Save checkpoint → modify files → restore → verify files match checkpoint

### Phase 2 — Review & Sync (Diff Review + Memory Sync)
**Dependency:** Phase 1 complete, at least one checkpoint saved
**Scope:** `diffReview.ts`, `syncPrompt.ts`, dashboard Diff Review + Memory Sync tabs
**Verification:** Modify a file manually → open Memory Sync → see file listed with correct diff stats

### Phase 3 — Memory (Persistent Rules)
**Dependency:** Python + TrueMemory library installed
**Scope:** `memory.ts`, `memory_bridge.py`, dashboard Parcel Memory tab
**Verification:** Add rule → restart IDE → verify rule persists → copy recall prompt

### Phase 4 — Dashboard Polish
**Dependency:** All phases complete
**Scope:** Overview, activity feed, real-time state sync, toast notifications
**Verification:** All 5 tabs load real data, no mocked/fake state shown

---

## 6. Do NOT Do

- Do not modify the agent's API or prompt injection directly
- Do not auto-commit on checkpoint restore (only restore working tree)
- Do not auto-reject all files — always let the developer decide per-file
- Do not store secrets or `.env` contents in the checkpoint database
- Do not block the developer if health check score is low — warn only
